# Backend Architecture & Deployment Guide

## 1. Architecture Overview
This project uses **Supabase** as the backend-as-a-service (BaaS) provider. The architecture is built around:
- **PostgreSQL Database**: Relational data with Row Level Security (RLS).
- **Supabase Auth**: User management with custom roles (Admin/Client).
- **Supabase Storage**: Secure object storage for galleries.
- **Edge Functions**: Serverless logic for payments, notifications, and complex operations.

## 2. Database Schema
The schema is defined in `schema.sql`. Key tables include:

### Core Identity
- `user_profiles`: Extends auth.users with `role` ('admin' | 'client').
- `clients`: CRM records managed by admins.

### Content
- `galleries`: Collections of photos with access control (`is_locked`, `is_paid`).
- `photos`: The actual media items. Stored in two variants: 'watermarked' and 'clean'.

### Commerce
- `packages`: Pricing tiers defined by admins.
- `payments`: Transaction records (M-Pesa).
- `sms_logs`: Audit trail for SMS communications.
- `admin_resources`: Tracks admin usage (SMS balance, storage).

### Security
- `audit_logs`: Immutable record of critical actions (deletions, exports).

## 3. Auth & Security Rules (RLS)
Security is enforced at the database level using Row Level Security (RLS).

- **Clients**: Can ONLY see records explicitly linked to them via `user_id` or `client_id`.
- **Admins**: Can see ALL records where they are the `owner_admin_id`.
- **Public**: No public read access to database tables.

### Storage Security
- `photos-watermarked`: Publicly readable (or restricted to authenticated users).
- `photos-clean`: **Private**. Access is granted only via short-lived signed URLs generated by the backend *after* verifying payment status.

## 4. Edge Functions (Logic Layer)
These functions reside in `supabase/functions/` and handle secure operations.

| Function | Purpose | Trigger |
|----------|---------|---------|
| `create_gallery` | Handles batch upload and DB creation | Admin Upload |
| `send_sms` | Dispatches SMS via provider | Notification Service |
| `initiate_payment` | Triggers M-Pesa STK Push | Client Payment |
| `payment_callback` | Validates M-Pesa webhook & updates DB | M-Pesa Server |
| `unlock_gallery` | Marks gallery paid & releases clean photos | Payment Success |
| `buy_sms_bundle` | Refills admin SMS balance | Admin Purchase |
| `analytics_sync` | Aggregates daily stats for dashboard | Cron Job |

## 5. Deployment Steps

### Prerequisites
1. Install Supabase CLI: `npm install -g supabase`
2. Login: `supabase login`

### Initial Setup
1. Link project: `supabase link --project-ref <your-project-id>`
2. Push DB Schema: `supabase db push` (uses `schema.sql`)

### Deploying Functions
```bash
supabase functions deploy create_gallery
supabase functions deploy initiate_payment
# ... deploy others
```

### Environment Variables
Set these in Supabase Dashboard > Settings > Edge Functions:
- `MPESA_CONSUMER_KEY`
- `MPESA_CONSUMER_SECRET`
- `SMS_PROVIDER_API_KEY`
- `ADMIN_SECRET_KEY` (if needed for elevated tasks)
